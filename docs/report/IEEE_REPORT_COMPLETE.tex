\begin{abstract}
StrategAI is a full-stack Civilization-style strategy game that integrates two generative AI capabilities in one playable system: Large Language Models (LLMs) for autonomous civilization behavior and Diffusion Transformers (DiTs) for on-demand visual asset generation. The backend constrains LLM output through a nine-intent tool-use interface, then lowers those intents into deterministic game actions handled by the same rule engine used for human players. The asset server uses a ComfyUI workflow around FLUX.2 Klein 4B Distilled to generate structures, units, terrain features, background tiles, objects, and leader images, with static and placeholder fallbacks when GPU inference is unavailable. A separate LoRA training pipeline adapts FLUX.2 Klein 4B toward a top-down medieval game-asset style using a curated 100-image synthetic dataset generated by the project team. The resulting system demonstrates practical integration patterns for combining language-model reasoning, image generation, model adaptation, and graceful degradation in a course-scale game project. We focus on verifiable architecture, implementation trade-offs, and limitations rather than claiming new model architectures or exhaustive quantitative benchmarks.
\end{abstract}

\begin{IEEEkeywords}
Large Language Models, Diffusion Transformers, Game AI, Asset Generation, LoRA Fine-Tuning, Tool Use, Multi-Agent Systems
\end{IEEEkeywords}

\section{Introduction}

Generative AI has created new opportunities for interactive entertainment, but most demonstrations remain limited to a single modality. A language model may generate dialogue, or a diffusion model may generate concept art, but integrating strategic reasoning, visual generation, user interaction, and runtime failure handling into one coherent game system remains an engineering challenge.

StrategAI addresses this challenge through a playable web strategy game. AI civilizations reason through an OpenAI tool-use chat model, while visual assets are generated by a self-hosted image pipeline. The project is deliberately system-oriented: it does not introduce a new LLM, diffusion model, sampler, or training algorithm. Its contribution is the architecture that makes these components work together without allowing probabilistic systems to corrupt deterministic game state.

\subsection{Motivation}

Independent and student game developers face two recurring problems when adopting generative AI. First, the most capable models are computationally expensive, particularly for image generation. Second, the operational failure modes are different from normal game code: a cloud LLM may be unavailable, a local GPU queue may be saturated, a generated asset may fail post-processing, or a model may produce an output that is aesthetically inconsistent with the rest of the game. A practical system must preserve gameplay even when AI services degrade.

StrategAI treats those constraints as first-class design requirements. The backend isolates LLM reasoning behind structured intents. The frontend renders color and glyph fallbacks when assets are missing. The asset server supports three generation modes: ComfyUI, static assets, and placeholder images. The training pipeline is separated from runtime gameplay, so fine-tuning can be performed offline and deployed as adapter weights.

\subsection{Contributions}

This report makes five contributions.

First, it presents an intent-based LLM integration layer for a rule-bound strategy game. The LLM chooses among nine strategic intent tools, while deterministic Python code performs pathfinding, validation, action lowering, and game-state updates.

Second, it documents a four-layer asset prompt architecture separating workflow configuration, style templates, semantic enum descriptions, and assembly logic. This makes prompt construction maintainable and testable across six asset families.

Third, it describes a three-stage leader portrait pipeline that generates splash art, profile portraits, and action scenes while reusing seeds and image-to-image references to preserve character identity.

Fourth, it reports a LoRA adaptation workflow for top-down medieval game assets, including a 100-image synthetic dataset, a six-experiment caption-detail by rank matrix, and qualitative checkpoint selection.

Fifth, it provides a reference implementation with documented backend, frontend, asset-server, and training-pipeline tests, while explicitly distinguishing verified claims from aspirational future work.

\section{Related Work}
\label{sec:related}

\subsection{Game AI and LLM Agents}

Traditional game AI relies on authored decision systems such as behavior trees~\cite{mark2012behavior}, utility systems~\cite{mark2015utility}, and domain-specific planners~\cite{rabin2013game}. These approaches are reliable and controllable, but require extensive authoring for each behavior. Recent LLM-based agents demonstrate richer behavior in simulated environments. Park et al.~\cite{park2023generative} explored social simulation with generative agents. Zhu et al.~\cite{zhu2023ghost} and Wang et al.~\cite{wang2023voyager} investigated LLM agents in Minecraft-like environments. Toolformer~\cite{schick2023toolformer} showed that language models can interact with external tools, motivating architectures where models emit structured calls instead of unconstrained text.

StrategAI follows this tool-use direction but applies it to a deterministic strategy-game engine. The LLM does not issue raw state mutations. It selects intents, and the game engine decides whether those intents are legal.

\subsection{Procedural Content Generation and Diffusion Models}

Procedural content generation has long been used in games, from Rogue~\cite{toy1980rogue} to large procedural worlds such as No Man's Sky~\cite{hello2016no}. Classical PCG includes noise functions~\cite{perlin1985image}, procedural building grammars~\cite{parish2006procedural}, and search-based design methods~\cite{smith2011answering}. Deep generative models expanded this space through GANs~\cite{goodfellow2014generative}, VAEs~\cite{kingma2013auto}, and diffusion models~\cite{ho2020denoising,rombach2022high}.

Diffusion Transformers replace the U-Net backbone with transformer blocks and have shown strong scaling behavior~\cite{peebles2023scalable}. Rectified-flow formulations further reduce sampling complexity~\cite{liu2023rectified}. StrategAI uses these advances through FLUX.2 Klein 4B Distilled, but the project contribution is not the model itself. The contribution is a game-facing service that wraps image generation with typed requests, deterministic prompt assembly, post-processing, caching, and fallbacks.

\subsection{LoRA and Model Adaptation}

Low-Rank Adaptation (LoRA) trains compact adapter matrices while keeping base model weights frozen~\cite{hu2022lora}. This makes model adaptation practical for student and independent developer settings, because the trained artifact is much smaller than a full model checkpoint. StrategAI uses LoRA to bind a top-down medieval visual style to a trigger token while preserving the base model for non-triggered generations.

\section{System Architecture}
\label{sec:architecture}

StrategAI consists of four subprojects: backend, frontend, asset server, and dataset/training pipeline. Each component has a different runtime profile and is isolated behind explicit interfaces.

\subsection{Backend}

The backend is a pure functional game engine implemented with frozen dataclasses. Actions return new game-state objects rather than mutating existing state in place. This design supports LLM integration because the model receives serialized views instead of direct access to internal state objects. The backend exposes 16 endpoints in total: 14 game endpoints plus health and audio endpoints.

\subsection{Frontend}

The frontend is a Next.js 15 and React 19 application. It renders the game world as an SVG hex map and uses TypeScript for API contract validation. State is held in component memory using React hooks such as \texttt{useState}, \texttt{useRef}, and \texttt{useMemo}; the current implementation does not use a persistent game-state store or persistent asset-manifest cache.

\subsection{Asset Server}

The asset server is a FastAPI service that exposes 35 REST endpoints across six asset families: structures, objects, terrain features, units, background tiles, and leaders. It supports three modes per family. In \texttt{comfyui} mode it delegates generation to ComfyUI~\cite{comfyui2024}. In \texttt{static} mode it serves pre-generated files. In \texttt{placeholder} mode it creates simple PIL-based images. This design allows the game to run even without a working GPU generation backend.

\subsection{Training Pipeline}

The training pipeline produces LoRA adapter weights for FLUX.2 Klein 4B using the Ostris AI Toolkit~\cite{ostris2024toolkit}. It uses a synthetic dataset generated and curated by the StrategAI team~\cite{stixxert2024dataset}. Training is offline and does not run during gameplay. The highlighted adapter, \texttt{strategai-lora-detailed-high-step1800.safetensors}, is 353 MB and is published with the other experiment variants~\cite{strategai2026topdownlora}.

\subsection{Data Flow}

During gameplay, the frontend sends player actions to the backend. The backend validates the action, updates immutable state, and returns a serialized view. When the player ends a turn, each AI civilization receives a local view of the game and emits strategic intents through the LLM tool-use interface. The engine lowers those intents into legal game actions.

Asset generation is initiated by the frontend when a new game or visual state requires sprites or portraits. The resolver maps game taxonomy to asset-server enums, requests generated or static assets, and falls back to built-in colors, glyphs, and initials when assets are unavailable.

\section{LLM-Driven Strategic AI}
\label{sec:llm}

The LLM layer controls three AI civilizations: Genghis Khan, Cleopatra, and Mahatma Gandhi. Each civilization shares the same intent interface but receives a different persona prompt.

\subsection{Intent-Based Abstraction}

Direct LLM control of game state is unsafe because spatial reasoning, arithmetic, and rule compliance must be exact. StrategAI therefore separates strategic choice from tactical execution. The LLM chooses a high-level intent such as \texttt{Expand}, \texttt{Engage}, or \texttt{Research}. Deterministic code then selects units, validates prerequisites, checks ownership, calculates distances, and applies the resulting action through the normal game engine.

\subsection{Intent Types}

The backend defines nine intent types:

\begin{enumerate}
  \item \texttt{Expand}: found or move toward founding a city.
  \item \texttt{Scout}: explore frontier tiles.
  \item \texttt{Engage}: attack or move toward an enemy civilization.
  \item \texttt{Reinforce}: defend friendly cities or positions.
  \item \texttt{Speak}: send a diplomatic message.
  \item \texttt{AdjustStance}: set peace, war, or alliance.
  \item \texttt{Build}: queue production.
  \item \texttt{Research}: select a technology.
  \item \texttt{Improve}: assign a worker to improve a tile.
\end{enumerate}

Each intent is implemented as a frozen dataclass. A simplified example is shown below.

\begin{lstlisting}[language=Python]
@dataclass(frozen=True, slots=True)
class Engage:
    """Wage war on another civilization."""
    target_civ_id: int
\end{lstlisting}

\subsection{Prompt and Memory Design}

The shared system prompt describes the game state schema, the available intent tools, and strategic guidelines. A persona prompt is appended for each leader. Genghis Khan is configured as aggressive and conquest-oriented, Cleopatra as diplomatic and opportunistic, and Gandhi as peaceful and research-focused.

The backend bounds LLM context by keeping the last 32 intents, the last 32 diplomatic messages, and an 8-turn recency window. This keeps prompts informative without allowing context size to grow unbounded over long games.

\subsection{Graceful Failure}

If the OpenAI API is unavailable or an LLM call fails, the backend returns no AI decisions for that civilization rather than crashing the game. Invalid or unresolvable intents are skipped or reported through machine-readable validation errors. This makes the LLM layer optional from the perspective of core game-state integrity.

\section{Diffusion Transformer Asset Pipeline}
\label{sec:asset}

The asset server translates typed asset requests into generated or fallback images. It uses FLUX.2 Klein 4B Distilled through ComfyUI for image generation and deterministic post-processing for game integration.

\subsection{Model and Runtime}

FLUX.2 Klein 4B Distilled is a 4-billion parameter Diffusion Transformer in the FLUX family~\cite{blackforestlabs2024flux}. The local ComfyUI workflow uses FP8 transformer weights and a larger text encoder; project documentation reports roughly 8.4 GB for the FP8 inference stack before runtime headroom. The complete model collection used by the workflow includes the transformer, text encoder, VAE, and optional StrategAI LoRA adapter.

The report avoids claiming universal runtime benchmarks because image generation speed depends on GPU model, ComfyUI configuration, queue state, resolution, and post-processing. The relevant architectural point is that generation is isolated behind an asset service and can be replaced by static or placeholder modes.

\subsection{Four-Layer Prompt Architecture}

Prompt construction is divided into four layers:

\begin{enumerate}
  \item Workflow configuration: ComfyUI JSON selects model nodes, sampler settings, resolution, and post-processing.
  \item Style templates: shared phrases define camera angle, pixel-art vocabulary, background expectations, and LoRA trigger tokens.
  \item Semantic descriptions: enum values map to hand-written prose for materials, structures, biomes, unit types, cultures, and moods.
  \item Assembly logic: typed request fields are validated and rendered into final prompts.
\end{enumerate}

This structure prevents prompt logic from being scattered across endpoints. It also makes tests possible at each layer: enum coverage, template fields, workflow patching, and generated request payloads.

\subsection{Asset Families}

The current engine outputs 128$\times$128 game assets for structures, objects, terrain features, units, and background tiles. Leader images are larger cinematic assets used in the diplomacy interface. The asset families have different constraints: structures and units need transparent backgrounds; background tiles need texture continuity; leader portraits prioritize visual identity over sprite compositing.

\subsection{Leader Portrait Pipeline}

Leader generation uses three stages. First, a splash image establishes the canonical leader appearance. Second, a profile portrait is generated from the splash reference for compact UI display. Third, an action scene uses image-to-image conditioning to place the same leader in a more dramatic composition. Seeds and generated metadata are stored so outputs can be traced and reproduced under the same model and workflow configuration.

\subsection{ComfyUI Integration}

The asset server communicates with ComfyUI through an asynchronous workflow lifecycle: upload inputs when needed, patch workflow JSON with prompt and seed values, queue the prompt, monitor execution through WebSocket with HTTP polling fallback, download outputs, and persist metadata. The server supports multi-node ComfyUI configurations and health checks, but all GPU work remains outside the web frontend and backend game engine.

\section{LoRA Fine-Tuning}
\label{sec:lora}

The base image model can sometimes produce top-down medieval assets from prompting alone, but occasional success is insufficient for an automated game pipeline. StrategAI therefore uses LoRA to make the top-down medieval style more reliable when a trigger token is present.

\subsection{Dataset Provenance}

The LoRA dataset is not built from OpenGameArt or other hand-authored game-art repositories. OpenGameArt is only used by the project for bundled music attribution in the frontend. The LoRA dataset was generated by the StrategAI team through its own ComfyUI-based pipeline using FLUX.2 [dev], Flux-2-Multi-Angles-LoRA-v2~\cite{lovis93_2024_multiangles}, PixelArt-Detector, and background-removal tooling. The resulting 100 images were manually curated for visual quality and top-down perspective.

The published dataset card describes a 100-image synthetic dataset consisting primarily of medieval structures: 97 structures, 2 vegetation assets, and 1 terrain asset. This skew is intentional because building sprites are the most important in-distribution target for the game. It also creates a limitation: generalization to units, terrain, and modern objects must be evaluated qualitatively rather than assumed from dataset balance.

\subsection{Caption Levels and Trigger Token}

Each image is paired with captions at three levels of detail:

\begin{itemize}
  \item Detailed captions: 50--100 words with material and architectural detail.
  \item Minimal captions: 20--30 words focused on the essential subject.
  \item Ultra-minimal captions: 5--10 words, often only style and viewpoint.
\end{itemize}

Training uses the \texttt{<tdp>} trigger token and the phrase \texttt{top-down view.}. At inference time, prompts containing this trigger activate the learned top-down medieval behavior. Prompts without the trigger can use the base model behavior, which is important for leader portraits and other assets that should not inherit the board-sprite viewpoint.

\subsection{Experiment Matrix}

The training pipeline evaluates a six-experiment matrix crossing caption detail with LoRA rank. High-rank variants use larger adapter capacity, while lower-rank variants compress the learned style more aggressively. The highlighted variant is \texttt{detailed-high} at step 1800, using detailed captions and rank settings documented in the published model card.

\subsection{Qualitative Findings}

The qualitative evaluation compares base model outputs with LoRA-conditioned outputs on in-distribution structures and out-of-distribution subjects. The base model may follow the phrase ``top-down view,'' but perspective is not reliable enough for unattended generation. The LoRA variants improve perspective consistency. The \texttt{ultra-low} variant generalizes strongly but sacrifices architectural detail. The \texttt{minimal-high} variant balances fidelity and generalization. The \texttt{detailed-high} variant gives the strongest in-distribution medieval structure fidelity and is therefore the deployment candidate for the game asset pipeline.

\subsection{License Implications}

The dataset and LoRA are published for academic demonstration with a non-commercial effective license because the dataset was generated through upstream FLUX.2 [dev] dependencies. This is separate from the repository code license. Any commercial reuse would require resolving the upstream model license chain.

\section{Frontend Integration}
\label{sec:frontend}

The frontend brings the backend and asset server together in a playable interface. It renders a hex map, panels for unit and city management, diplomacy UI, research controls, and generated leader imagery.

\subsection{Asset Manifest Resolution}

The frontend maps game concepts to asset-server vocabulary deterministically. Thirteen terrain types map to four background tile types: grass, stone, sand, and water. Seven game unit types map to four unit asset types: warrior, archer, scout, and settler. Leader names map to asset-server archetype and culture parameters. For the current roster, Genghis Khan maps to \texttt{warrior\_king} and \texttt{east\_asian\_imperial}; Cleopatra maps to \texttt{diplomat} and \texttt{ancient\_egyptian}; Mahatma Gandhi maps to \texttt{philosopher\_king} and \texttt{south\_asian}.

The current frontend resolves manifests fresh at campaign start. It removes old \texttt{inf3600:assetManifest:*} localStorage keys left by earlier versions, but does not use persistent manifest caching. This avoids stale image URLs when game identifiers are reused.

\subsection{Fallback Behavior}

Fallback behavior is intentionally layered. If a generated terrain image is missing, the map uses a color fill. If a unit sprite is missing, the UI renders a colored circle and initial. If a structure sprite is missing, the map uses built-in markers. If leader portraits are unavailable, the diplomacy UI displays initials. These fallbacks preserve playability even when the asset server, ComfyUI, or GPU backend is unavailable.

\subsection{Diplomacy UI}

The diplomacy interface supports free-text messages and typed diplomatic actions such as threats, peace offers, declarations of war, and alliance proposals. AI responses are generated through the backend diplomacy layer and conditioned by the same leader persona design used for strategic actions.

\section{Evaluation}
\label{sec:evaluation}

The evaluation focuses on verifiable implementation properties and qualitative behavior, avoiding unsupported benchmark or coverage claims.

\subsection{Testing}

The backend test suite collected locally contains 339 tests covering game engine logic, API behavior, serialization, diplomacy, fog-of-war behavior, intent resolution, combat, production, and turn progression. The asset-server testing plan documents 547 tests covering API endpoints, prompt assembly, workflow patching, database behavior, caching, load balancing, and concurrency. The training pipeline README documents 35 tests for caption generation, dataset validation, configuration syncing, and pipeline execution. The frontend uses TypeScript strict mode; no dedicated frontend test runner is configured in \texttt{package.json}.

\subsection{Runtime Behavior}

The backend game engine is lightweight enough for interactive local play. LLM turn time is dominated by external API latency and prompt size, not by game-state resolution. Asset generation latency is dominated by ComfyUI and GPU queue behavior. Static and placeholder modes provide fast non-GPU alternatives. The frontend bounds concurrent asset requests so the browser does not flood the asset server.

\subsection{AI Behavior}

Qualitative playtesting shows persona-differentiated behavior. Genghis Khan tends toward expansion and warfare, Cleopatra toward diplomacy and opportunistic alliances, and Gandhi toward research and defense. The memory window helps leaders respond to recent messages and avoid immediate repetition. Limitations remain: the agents can repeat weak strategies, place cities suboptimally, and struggle with coordinated multi-unit tactics.

\subsection{Asset Quality}

Generated assets are strongest for in-distribution medieval structures, which match the dataset skew. Top-down perspective improves when the LoRA trigger is used. Leader identity is preserved well enough for the prototype through the three-stage image-to-image flow. Generalization to asset types underrepresented in the dataset is useful but less reliable, especially for complex modern or mechanical subjects.

\section{Ethical and Legal Considerations}
\label{sec:ethics}

\subsection{Bias and Representation}

The project uses historical figures as strategic archetypes. This carries a risk of reductive cultural representation. The prompts attempt to frame each leader respectfully: Genghis Khan as disciplined and conquest-oriented rather than barbaric, Cleopatra as intelligent and diplomatic rather than exoticized, and Gandhi as principled rather than passive. Prompting alone is not a complete bias mitigation strategy. The system does not include a dedicated bias detector or moderation model, and prior work has shown that large language models can reproduce and amplify biases present in training data~\cite{bender2021stochastic}.

The image pipeline has a related representational limitation. The LoRA dataset is synthetic and heavily structure-focused, with a medieval architectural vocabulary. This can create stylistic mismatch when representing non-European or non-structure assets. Leader portraits depend on the base model's learned distribution and may reproduce biases present in large image datasets~\cite{crawford2019excavating}.

\subsection{Content Moderation}

Diplomacy accepts free-form player input and returns LLM-generated text. The current system relies on system prompts and leader personas rather than a separate moderation layer. This is acceptable for a controlled course prototype, but not sufficient for a public deployment. A production version should add input and output moderation before displaying model-generated text to players.

\subsection{Environmental Cost}

Image generation and cloud LLM inference consume energy~\cite{strubell2019energy}. StrategAI reduces unnecessary computation through bounded LLM memory, smaller image models, FP8 inference workflows, and static or placeholder modes. The report does not estimate per-image energy because no instrumented power measurement was performed for this submission.

\subsection{Copyright and Licensing}

The legal status of AI-generated content remains unsettled~\cite{samuelson2023generative}. StrategAI has three relevant layers. First, the code repository may be open source, but that does not automatically make all model-derived assets commercially usable. Second, the LoRA dataset and adapter inherit non-commercial restrictions from upstream FLUX.2 [dev] dependencies. Third, OpenGameArt is used only for music attribution in the frontend, not for the LoRA image dataset. The project therefore should be described as an academic demonstration unless and until the full model and dataset license chain is cleared for commercial use.

\subsection{Reproducibility}

The asset server stores generated seeds and uses \texttt{secrets.randbits(32)} when no seed is provided. Exact reproduction still requires the same model weights, workflow graph, ComfyUI nodes, post-processing settings, and software versions. LLM behavior is less reproducible because the model is accessed through an external API; provider-side updates can change behavior even with identical prompts. The intent interface limits the shape of possible actions, but not the model's strategic choices.

\section{Discussion and Limitations}
\label{sec:discussion}

\subsection{Architectural Trade-offs}

The microservice split separates CPU-bound game logic from GPU-bound image generation, but increases operational complexity. The intent layer makes AI behavior safer and testable, but reduces the LLM's freedom to invent new actions. The fallback-heavy asset system increases code paths, but makes the game usable across development machines without assuming a working GPU pipeline.

\subsection{Technical Limitations}

The backend currently uses in-memory state, which limits deployment to a single process. Persistent storage would be required for robust multiplayer or long-running hosted games. Asset generation is synchronous from the API client's perspective; a production service should use a job queue with progress updates. The frontend does not yet have a dedicated automated test suite. The LoRA dataset is small and skewed toward structures, limiting claims about broad asset generalization.

\subsection{Future Work}

Future work should add persistent storage, asynchronous asset jobs, stronger frontend tests, quantitative asset evaluation, moderation for diplomatic chat, and locally hosted LLM alternatives for better reproducibility. The asset pipeline could be extended to animation, sound effects, UI icons, and user-defined civilizations, but those features should be introduced only with clear licensing and evaluation procedures.

\section{Conclusion}
\label{sec:conclusion}

StrategAI demonstrates that LLM-driven strategy behavior and DiT-based asset generation can be integrated into a coherent game prototype when probabilistic components are isolated behind deterministic interfaces. The key engineering patterns are intent lowering for LLM actions, typed prompt assembly for image generation, offline LoRA adaptation for visual consistency, and graceful degradation throughout the frontend and asset pipeline.

The project is strongest as a reference architecture for generative-AI game integration. Its main limitations are also clear: single-instance backend state, synchronous generation, external LLM dependency, small LoRA dataset, and non-commercial model-derived training artifacts. By documenting both the working system and these constraints, StrategAI provides a realistic foundation for future work rather than an inflated benchmark claim.

\section{Endpoint Summary}

The backend exposes 16 endpoints: 14 game endpoints plus \texttt{/health} and \texttt{/audio/intro}. The asset server documents 35 REST endpoints across global operations and six asset families. The frontend communicates with both services through typed client wrappers and degrades to built-in visual fallbacks when asset requests fail.
